SATURN ↔ MARS

secure relay · outbound-only · zero-persist · zero-buffer

Topology

● request / ACK flow Saturn: RAM-only hop, metadata logs only Mars: ACK only after fsync commit Keep-alive: PING 5s / timeout 10s Inbound to Mars: all blocked

Zero-buffer rule — Saturn holds nothing for Mars

One payload, one hop, one tenant — never a queue

Saturn never accumulates Mars-bound data. There is no queue, no replay buffer, no store-and-forward, no batching across requests. A payload exists in Saturn's RAM only for the duration of a single active hop, and the per-tenant in-flight cap bounds even that. If Mars cannot take the payload right now, the request is rejected immediately — the third party owns the retry, Mars's corr-id dedupe makes the retry safe.

IF registry[mars-id] ≠ ONLINE → reject 503, drop nothing because nothing was held
IF in-flight[mars-id] ≥ CAP (default 1) → reject 429, no second payload enters RAM
IF stream closes mid-hop → cancel waiter, reject 502, payload freed — not parked

SATURN MAY HOLD (RAM, metadata)

  • corr-id → response waiter (in-flight only)
  • mars-id → stream handle + state
  • disconnect timestamps, health telemetry
  • the one payload currently mid-hop, per tenant cap

SATURN NEVER HOLDS

  • queued / pending payloads for an offline Mars
  • multiple buffered payloads per Mars beyond cap
  • payloads on disk, DB, or persistent logs
  • copies for retry, replay, or recovery

Mars down

01MARSDC link drops — silent failure, no FIN
02SATHTTP/2 PING unanswered → dead in ≤10s → registry OFFLINE
